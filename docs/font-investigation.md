# Font Investigation — Status (RESOLVED)

## Current state (as of 2026-07-06)

**All fonts confirmed rendering correctly in PDF output.** The root cause was
the fontsource subset files (14KB IBMPlexMono-400, 15KB IBMPlexMono-600) being
too stripped down — Chromium loaded the font but silently fell back to system
monospace for glyph rendering because the subsets lacked ASCII/digit glyphs.

The fix: replaced the 14KB/15KB fontsource subsets with full 49KB/50KB woff2
files from the official IBM Plex repository `@ibm/plex-mono@2.5.0`.

### Files in src/assets/fonts/

| File | Size | Source | Glyph coverage |
|------|------|--------|---------------|
| IBMPlexSans-Variable.woff2 | 45KB | @fontsource-variable/ibm-plex-sans | Full (variable, all weights) |
| IBMPlexMono-400.woff2 | 49KB | IBM Plex repo (complete) | Full — replaces 14KB subset |
| IBMPlexMono-600.woff2 | 50KB | IBM Plex repo (complete) | Full — replaces 15KB subset |
| JetBrainsMono-Regular.woff2 | 90KB | github.com/JetBrains/JetBrainsMono | Full (unused, available for future) |
| Urbanist-Variable.woff2 | 27KB | @fontsource-variable/urbanist | Full (variable, all weights) |

### How the fix was confirmed

Used Canvas `measureText()` to compare per-character widths at 40px:

| Char | IBM Plex Mono 400 | system monospace | Verdict |
|------|-------------------|------------------|---------|
| `0`  | 20.00px           | 24.00px          | Different |
| `W`  | 37.75px           | 24.00px          | Different |
| `i`  | 11.11px           | 24.00px          | Different |

Before the fix, all characters measured 24.00px (identical to system fallback).
After the fix, each character has distinct IBM Plex Mono metrics.
